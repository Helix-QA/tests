# Описание автотеста
**«Формирование чеков коррекции при различных способах оплаты и применении НДС»**
## Цель теста

Проверить корректность формирования **чеков коррекции** в системе при различных условиях оплаты и налогообложения, а именно:

* правильное формирование **XML чека коррекции**;
* соответствие данных **формы пользователя данным XML**;
* корректную работу:
  * признака способа расчёта,
  * типа оплаты,
  * НДС;
* корректную логику формирования **пары чеков коррекции** (сторно + корректный чек);
* независимость сценариев и восстановление исходного состояния системы.

## Общие условия выполнения
Перед выполнением каждого сценария:
* очищаются все временные переменные;
* создаётся тестовый клиент;
* создаётся номенклатура **«Услуга_Групповая»**;
* фиксируется текущее время для последующих проверок печатных форм и XML.
Каждый сценарий является **самодостаточным** и не зависит от результатов предыдущих.

## Общая логика формирования чеков коррекции

При создании чеков коррекции система формирует **два чека**:

1. **Сторнирующий чек**
   (`OperationType = 2`) — отменяет ошибочную фискальную операцию.
2. **Корректирующий чек**
   (`OperationType = 1`) — фиксирует корректные данные.
Оба чека проверяются отдельно.

## Проверка состава XML чека коррекции
В рамках теста валидируется:
### Блок `<Parameters>`
* корректный тип операции (`OperationType`);
* тип коррекции:
  * `Type = 0` — самостоятельная коррекция;
  * `Type = 1` — коррекция по предписанию;
* наличие и корректность описания причины коррекции;
* передача дополнительного реквизита.

### Блок `<Positions>`
Для позиции **«Услуга_Групповая»** проверяется:
* количество = `1`;
* сумма = `250`;
* признак предмета расчёта = **Услуга**;
* признак способа расчёта = **Передача с полной оплатой**
  (`PaymentMethod = 4`);
* корректная передача ставки НДС:
  * при включённом НДС — `VATRate = 10`, `VATAmount = 22.73`;
  * при выключенном НДС — `VATRate = none`.

### Блок `<Payments>`
Проверяется корректное распределение суммы оплаты в зависимости от источника средств:
* при оплате наличными:
  ```xml
  Cash = 250
  ```
* при оплате лицевым счётом:
  ```xml
  PrePayment = 250
  ```
Остальные поля оплаты во всех случаях равны `0`.

## Проверка соответствия UI и XML
После формирования чеков выполняется сравнение:

### Таблица **«ПозицииЧека»**
* наименование услуги;
* количество;
* сумма;
* ставка и сумма НДС;
* признак способа расчёта.

### Таблица **«ТаблицаОплат»**
* тип оплаты;
* сумма оплаты.
📌 Данные таблиц **полностью соответствуют XML**, что подтверждает корректность отображения фискальных данных пользователю.

## Сценарий оплаты через лицевой счёт
В рамках теста дополнительно проверяется сценарий, при котором:
1. Клиент предварительно пополняет лицевой счёт.
2. Продажа оплачивается за счёт средств лицевого счёта.
3. При формировании чека коррекции:
   * источник средств отражается как **Предоплата** (`PrePayment`);
   * признак способа расчёта остаётся **Передача с полной оплатой**, так как услуга оказана полностью.
Это подтверждает корректное разделение понятий:
* **источник денег** (тип оплаты),
* **экономический смысл расчёта** (PaymentMethod).

## Проверка работы НДС
В отдельном сценарии:
1. В настройках организации устанавливается ставка НДС **10%**.
2. Создаётся продажа и формируется чек коррекции.
3. Проверяется:
   * корректный расчёт суммы НДС (`22,73`);
   * отображение НДС в XML и в форме.
4. После завершения теста ставка НДС возвращается в состояние **«Без НДС»**.
Это исключает влияние сценария на последующие тесты.

## Ожидаемый результат
* XML чеков коррекции формируется корректно;
* данные формы полностью соответствуют XML;
* тип коррекции и тип операции определяются правильно;
* НДС рассчитывается и передаётся корректно;
* тип оплаты не влияет на признак способа расчёта;
* система корректно формирует пару чеков коррекции;
* состояние системы восстанавливается после выполнения теста.
