### Общее назначение файла
**Функционал: Пациент**
Этот сценарий проверяет создание карточки пациента и работу механизма **автоматического изменения статусов пациента**.
Файл состоит из двух основных частей:
1. Полноценное создание пациента со всеми возможными данными и проверка всех элементов карточки.
2. Проверка логики смены статусов пациента («Потенциальный» → «Новый» → «Постоянный» → «Невернувшийся» → «Отказ»).

### Структура сценариев
**Сценарий: Пациент** (основной и самый большой)
**1. Подготовка переменных**  
Генерируются случайные данные, чтобы каждый запуск теста создавал уникального пациента:
- ФИО (Фамилия + Имя + Отчество)
- Дата рождения, ИНН, СНИЛС, реестровый номер ОМС
- Email, номер телефона, данные для мобильного приложения
- Паспортные данные (серия, номер, код подразделения, дата выдачи)

**2. Создание элемента (карточки пациента)**  
Выполняется полное заполнение карточки пациента:
- Основные данные: ФИО, дата рождения
- Контактная информация: Email, телефон, мобильное приложение, социальная сеть
- Удостоверение личности (паспорт РФ)
- ИНН, СНИЛС, место работы
- Добавление медицинской карты
- Добавление родственника (мать) с общими лицевым, бонусным и накопительным счетами
- Добавление полиса, дисконтной карты, скидки, лицевого и бонусного счёта
- Социальная группа и социальный статус
- Маркетинговая информация: куратор, источник информации, канал привлечения
- Медицинские показания: аллергия, группа крови
- Дополнительная информация и комментарий

**3. Проверка всех возможностей карточки пациента**  
Тест проходит почти по всем кнопкам и вкладкам:
- Добавление предварительной записи, записи на приём, в лист ожидания, плана лечения
- Добавление задач (звонок входящий/исходящий, онлайн-заявка, прочее)
- Добавление заметки, SMS/Push
- Создание сделок («Новый пациент» и «Постоянный пациент»)
- Печать справок, наряд-заказов, гарантийного талона, квитанции
- Проверка всех вкладок: Взаиморасчёты, Лицевой счёт, Приёмы, Зубная формула, Планы лечения, Оценки и отзывы, Сегменты, Гарантии, Диалоги, Договоры, Наряд-заказы, История звонков, Снимки, Файлы и т.д.
- Сохранение и закрытие карточки пациента

**Сценарий: 02. Статус пациента**
Этот сценарий специально проверяет логику автоматического присвоения статусов пациенту.

**Что проверяется:**
- **Потенциальный** — сразу после создания пациента (без приёмов)
- **Новый** — после первого завершённого приёма
- **Постоянный** — после второго и более завершённых приёмов (вызывается регламент `АктуализироватьСтатусыПациентов()`)
- **Невернувшийся** — пациент не приходил больше периода, указанного в настройках параметров учёта (в тесте ставится 1 день)
- **Отказ** — статус присваивается только при явном переводе сделки потенциального пациента в отказ

Для каждой проверки:
- Создаётся новый пациент
- Выполняются нужные действия (приёмы с разными датами)
- Вызывается регламентное задание для обновления статусов
- Проверяется, что на форме пациента в полях статуса отображается правильное значение


### Ключевые особенности теста
- Используются только случайные данные — тест можно запускать много раз подряд без конфликтов
- Проверяется почти весь интерфейс карточки пациента (кнопки, вкладки, печать, задачи)
- Тщательно тестируется бизнес-логика смены статусов пациента
- Используется серверный код для принудительного запуска регламента `АктуализироватьСтатусыПациентов()`
- Проверяются как основные поля статуса, так и шаблон статуса (`ПолеСтатус_Шаблон`)

### Краткое резюме для памяти
Этот файл **полностью тестирует карточку пациента**: от создания со всеми данными и проверкой всех кнопок и вкладок — до автоматической смены статусов («Потенциальный → Новый → Постоянный → Невернувшийся → Отказ»).

Если через время откроешь файл — просто вспомни:  
«Создаём полноценного пациента + проверяем, как меняются его статусы в зависимости от приёмов и времени».
